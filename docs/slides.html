MCP Tool: get_dosen_pembimbing


@mcp.tool()
def get_dosen_pembimbing(nama_mahasiswa: str) -> str:
    """
    Use this tool to find the academic advisor (dosen pembimbing)
    of a student.

    Args:
        nama_mahasiswa: Full name of the student

    Returns:
        Name of the academic advisor
    """
    query = """
        SELECT d.nama_dosen
        FROM mahasiswa m
        JOIN dosen d ON m.id_dospem = d.id
        WHERE m.nama_mahasiswa = ?
    """
    result = db.execute_query(query, (nama_mahasiswa,), fetch_one=True)

    if result:
        return result[0]
    else:
        return f"Student {nama_mahasiswa} not found"

3 Hal Penting dalam MCP Tool

1. Docstring yang Jelas üìù

AI membaca docstring untuk memahami fungsi tool

2. Parameterized SQL ‚ö†Ô∏è


# ‚úÖ AMAN - Parameterized
cursor.execute("WHERE nama = ?", (nama_mahasiswa,))

# ‚ùå BAHAYA - SQL Injection!
cursor.execute(f"WHERE nama = '{nama_mahasiswa}'")

3. SQL Ditulis oleh MANUSIA üë®‚Äçüíª

AI tidak pernah generate SQL sendiri!

Mengapa Parameterized SQL?

Contoh serangan SQL Injection:


nama_mahasiswa = "'; DROP TABLE mahasiswa; --"

# Dengan f-string (BAHAYA!):
query = f"WHERE nama = '{nama_mahasiswa}'"
# Result: WHERE nama = ''; DROP TABLE mahasiswa; --'
# üí• Tabel mahasiswa terhapus!

# Dengan parameterized (AMAN!):
cursor.execute("WHERE nama = ?", (nama_mahasiswa,))
# Parameter diperlakukan sebagai literal string
# ‚úÖ Aman dari injection

Key Message #2

"SQL written by humans, not AI."

This is security by design.

RAG-for-Tools System

‚è±Ô∏è 10 menit

The Scaling Problem

Bayangkan punya 1000 tools

Apakah AI harus baca semua deskripsi 1000 tools
setiap kali user bertanya?

Masalah:

  • ‚ùå Melebihi token limit (context window)
  • ‚ùå Lambat (banyak text yang diproses)
  • ‚ùå Mahal (token = uang)
  • ‚ùå AI bingung (information overload)

Solusi: RAG-for-Tools

‚ùå Tanpa RAG

Agent lihat 1000 tools

  • Token limit exceeded
  • Lambat & mahal
  • AI overwhelmed

‚úÖ Dengan RAG

Agent lihat 3 tools (top relevant)

  • Cepat & murah
  • Fokus & akurat
  • Scalable!

Cara Kerja RAG

Step 1: Indexing (Setup Awal)


# Load tool descriptions
tools = load_json("tool_descriptions.json")

# Convert to embeddings (vectors)
text = "Get current date and time Jakarta timezone..."
embedding = model.encode(text)
# Result: [0.12, -0.34, 0.56, ..., 0.23]  (384 numbers)

# Store in ChromaDB
chromadb.add(embeddings, metadata=tools)

Step 2: Retrieval (Setiap Query)


# User query
query = "Jam berapa sekarang?"

# Find similar tools
results = chromadb.query(query_embedding, top_k=3)
# Returns: get_waktu_saat_ini (similarity: 0.82)

Similarity Score Explained

Cosine Similarity:


similarity = (query_vector ¬∑ tool_vector) /
             (|query_vector| √ó |tool_vector|)

0.0 = Completely different

0.5 = Somewhat similar

1.0 = Identical

Threshold biasanya > 0.3

Architecture Decisions

Why GPT-4o-mini?

  • Fast (~500-1000ms)
  • Cheap (~$0.0002 per query)
  • Good enough for tool selection

Why Temperature = 0?

  • Deterministic responses
  • Same query ‚Üí same answer
  • Good for production

Why Top-K = 3?

  • Balance: enough options vs context size
  • Sweet spot: K=3-5

Performance Metrics

Component Time Cost
RAG Retrieval ~60ms $0
LLM Decision (GPT-4o-mini) ~800ms $0.0002
Tool Execution ~10ms $0
Total per Query ~870ms $0.0002

1000 queries = ~$0.20 üí∞

Scalability

Number of Tools RAG Time Cost
10 tools ~60ms Same
100 tools ~60ms Same
1000 tools ~80ms Same

‚úÖ Performance stays constant!

Add 1000 tools, performance stays the same.
